# STM32 + ESP32 最小 HIL 半实物仿真进展记录

## 1. 阶段目标

在原有 STM32 计数与电机控制系统基础上，使用 ESP32-S3 作为虚拟传感器 / 轻量级仿真平台：

- 通过 GPIO 脉冲向 STM32 PB14 注入外部计数输入
- 通过串口读取 STM32 状态反馈
- 根据 STM32 当前状态自动决定是否发送脉冲

实现最小硬件在环测试闭环。

## 2. 系统角色划分

| 组件 | 角色 | 说明 |
|------|------|------|
| STM32F103 | 真实控制器 / DUT | 运行计数状态机 + 电机控制 + 状态输出 |
| ESP32-S3 | 虚拟传感器 / 仿真平台 | 生成脉冲、读取状态、自动调节 |
| PB14 | 计数输入注入点 | 原本接红外传感器，现由 ESP32 GPIO4 注入脉冲 |
| ESP32 GPIO4 | 虚拟传感器脉冲输出 | 模拟红外传感器 DO 数字输出 |
| STM32 PB10 (USART3_TX) | 状态反馈输出 | 周期性输出 `STATE=...` 状态行 |
| ESP32 GPIO18 | 状态反馈读取 | UART1 RX，接收并解析 STM32 状态 |
| OLED / 串口日志 | 状态观察与调试 | 本地 OLED + ESP32 串口监视器 + 网页显示 |

## 3. 硬件连接

```
ESP32 GPIO4  ──[1kΩ]──→  STM32 PB14     # 虚拟传感器脉冲注入
ESP32 GND    ──────────→  STM32 GND      # 共地
STM32 PB10   ──────────→  ESP32 GPIO18   # 状态反馈（UART RX）
```

注意事项：

- GPIO4 到 PB14 串联 1kΩ 电阻，防止 GPIO 配置错误时两个输出引脚短路
- 两块板各自 USB 供电，不连接 3.3V
- ESP32 GPIO4 配置为推挽输出，高电平 ≈ 3.3V，低电平 ≈ 0V
- STM32 PB14 配置为 IPU（内部上拉），与红外传感器输入配置一致
- 波特率 115200 8N1

## 4. 已实现功能

### 4.1 虚拟传感器脉冲注入

ESP32 通过串口命令控制 GPIO4 输出脉冲，模拟红外传感器被遮挡时的上升沿信号。

### 4.2 测试命令

| 命令 | 功能 |
|------|------|
| `once` | 手动输出 1 次虚拟传感器脉冲 |
| `burst N` | 输出 N 次脉冲（不检查 STM32 状态） |
| `slow` | 低频率持续输出，用于观察 STM32 逐次计数 |
| `normal` | 正常频率持续输出 |
| `fast` | 高频率持续输出 |
| `interval N` | 调整脉冲间隔为 N ms |
| `stop` | 停止所有脉冲输出 |
| `auto` | 自动模式：根据 STM32 状态决定是否输出脉冲 |

### 4.3 STM32 状态反馈读取

ESP32 通过 UART1 (GPIO18) 接收 STM32 USART3 输出的 `STATE=...` 状态行，解析出：

- STATE：IDLE / COUNTING / MOTOR_RUN / DONE
- TARGET：目标计数值
- CURRENT：当前计数值
- TOTAL：累计计数值
- MOTOR：ON / OFF

### 4.4 已验证的完整流程

- ESP32 手动输出单次虚拟传感器脉冲 ✅
- STM32 正确响应注入的 PB14 脉冲，CURRENT 计数 +1 ✅
- ESP32 读取 STM32 串口状态行并正确解析 ✅
- auto 模式下，ESP32 根据 STM32 状态自动控制脉冲启停 ✅
- STM32 进入 IDLE / DONE 后 ESP32 停止输出脉冲 ✅
- burst N 模式下准确输出指定数量的脉冲 ✅

### 4.5 RESET 双向控制

- ESP32 网页 RESET 按钮 → UART TX → STM32 USART3 RX → 复位到 IDLE ✅
- ESP32 串口命令 `reset` → 同样触发 STM32 复位 ✅

## 5. 关键问题与定位过程

### 问题：once 命令导致一次脉冲计数两次

**现象**：

发送 `once` 命令后，STM32 CURRENT 有时 +2。

**定位过程**：

1. 使用逻辑分析仪观察 PB14 波形，确认每次 once 只产生了一个 LOW → HIGH 边沿
2. 确认 STM32 PB14 侧 EXTI 配置为 `EXTI_Trigger_Rising`（仅上升沿触发），排除了双边沿触发的可能性
3. 推断问题不在 STM32 侧，而在 ESP32 once 命令逻辑可能产生了两次有效上升沿
4. 检查代码发现 once 命令执行时，`forceOnePulse` / `sendOnePulse` 可能与 `burstRemain` 或 `manualRun` 状态重叠，导致同一周期内输出了两次 LOW → HIGH 跳变

**根因**：once 命令与正在运行的 burst/manual 模式产生了两次触发。

**修复**：once 执行前关闭 `manualRun`，清空 `burstRemain`，确保只产生一次有效的 LOW → HIGH 上升沿。

**当前状态**：已修复，once 命令稳定每次计数 +1。

## 6. 当前工程意义

该系统已经从普通 STM32 控制项目，升级为具备最小 HIL（Hardware-in-the-Loop）思想的控制器测试平台。

HIL 基本闭环在本系统中的体现：

```
仿真平台输出激励 → 真实控制器响应 → 状态反馈 → 仿真平台调整激励

ESP32 GPIO4 脉冲  →  STM32 状态机+电机  →  STATE=...  →  ESP32 auto 模式决策
```

- **仿真平台（ESP32）**：生成虚拟传感器信号，模拟外部物理环境输入
- **真实控制器（STM32）**：运行状态机，响应输入，控制电机，反馈状态
- **状态反馈（UART）**：仿真平台根据控制器状态决定下一步输入

相比纯软件仿真，HIL 的关键特点是**控制器是真实硬件**，ESP32 替代的是传感器和外部环境，而不是替代控制器本身。

## 7. 后续计划

### 7.1 故障注入模式

| 模式 | 说明 |
|------|------|
| `stuck_high` | 模拟传感器输出卡在高电平，PB14 持续 = 1 |
| `stuck_low` | 模拟传感器输出卡在低电平，PB14 持续 = 0 |
| `recover` | 从 stuck 模式恢复，恢复正常脉冲 |
| `drop N` | 模拟传感器漏检：发送 N 次脉冲但随机丢失 M 次 |
| `noise N` | 模拟误触发：在正常脉冲之间注入 N 个随机干扰脉冲 |
| `jitter on/off` | 模拟传送速度波动：脉冲间隔加入随机抖动量 |

### 7.2 执行器链路闭环（IMU → STM32 → 舵机/电机）

- 接入 MPU6050 / ICM-42688 IMU 模块（I2C），作为真实传感器输入
- 接入 SG90 舵机，作为执行器输出
- 形成 IMU 角度 → STM32 控制逻辑 → 舵机角度 / 电机速度 的完整闭环
- 将现有电机控制经验扩展到多执行器

### 7.3 Linux 日志集成

- Linux 端自动记录每次测试的脉冲序列和 STM32 状态变化
- 生成测试报告（脉冲数、状态转换时间点、异常事件）

## 8. 当前状态结论

当前 STM32 + ESP32 最小 HIL 仿真雏形已经跑通：

- ESP32 可以模拟外部计数传感器，向 STM32 PB14 注入可控脉冲
- auto 模式下，ESP32 根据 STM32 状态自动控制脉冲启停
- 实现了"真实控制器 + 虚拟传感器输入 + 状态反馈"的基本 HIL 闭环
- RESET 双向控制已打通（网页 → ESP32 → UART → STM32）
- 故障注入模式已规划，为下一步扩展做好准备
